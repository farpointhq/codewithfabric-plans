Per-Member Cap Enforcement — Current vs Target Data Flow
Why TeamMember.monthlyLimitCents is theater today and what wiring it looks like after step 6
Database (Prisma) Next.js application LiteLLM proxy (enforcement layer)
CURRENT — broken
Pre-step-6
TeamMember.monthlyLimitCents
Saved by PATCH /api/team/members/[id]. Read in dashboard UI. Value: 5000 (=$50).
↓ not synced
litellm.generateKey / updateKey
Call sites never pass maxBudget. Keys are born with max_budget: null.
↓ not pushed
LiteLLM proxy — key record
max_budget: null
soft_budget: null
budget_duration: null
↓ never enforced
POST /api/litellm/callback (post-hoc)
Increments currentMonthSpendCents. Emits console.warn("Member X exceeded monthly limit"). Request is already served.
Failure mode
A member with a $50/mo cap can burn $5,000. Cap is a UI label, not a gate. The proxy has no idea the cap exists.
TARGET — wired
After step 6
TeamMember.monthlyLimitCents
Same Prisma write. Still the source of truth team owners edit through the dashboard.
↓ sync block on write
litellm.generateKey / updateKey
Pass maxBudget: cents / 100. New sync block in PATCH members/[id] mirrors the existing rateLimitRpm block. Fail-open on LiteLLM errors.
↓ stored on key
LiteLLM proxy — key record
max_budget: 50.00
soft_budget: 40.00 (80% warn)
budget_duration: "30d"
↓ enforced at request time
_PROXY_MaxBudgetLimiter (already loaded)
Existing LiteLLM v1.82.2 success callback. Blocks over-budget requests at the proxy, before vLLM is touched.
Enforcement path
When a member's spend ≥ max_budget, the proxy returns a 4xx. Next.js catches it, translates to a graceful "quota reached" UX. Caps become real.
