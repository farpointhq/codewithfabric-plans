Step 6 — Three-Stage Rollout Plan
Every stage is reversible via env var. Each gate is a hard checkpoint — we do not advance until the previous stage has proven safe.
STAGE 1
Wiring & Backfill
Plumbing only. No enforcement. Ships to anyone, any time.
What changes
  • Extend lib/litellm.ts with softBudget, budgetDuration params
  • Pass maxBudget in generateKey at invite-accept + key-generate
  • Add maxBudget sync in PATCH members/[id] mirroring rateLimitRpm
  • Idempotent backfill script
  • LITELLM_BUDGET_MODE=off
NET EFFECT
Every push is null because nobody has caps set. Pure scaffolding. Zero user-facing behavior change.
1→2
Plumbing green · test invite works
STAGE 2 — SHADOW
soft_budget only
Alerts fire. Nothing blocks. LiteLLM-native shadow mode.
What changes
  • Set LITELLM_BUDGET_MODE=shadow
  • Code pushes soft_budget: cents/100, leaves max_budget: null
  • New POST /api/litellm/budget-alerts webhook receiver
  • New BudgetAlert Prisma model for alert history
  • Slack #billing-alerts + dashboard widget for team owners
NET EFFECT
Test member (Ryan, $1 cap) generates would-have-blocked alerts. Real members with real caps produce telemetry. Zero user blocks.
2→3
Min 3 days · zero false positives · Ryan sign-off
STAGE 3 — ENFORCE
max_budget flip
Real enforcement. Ryan first, then open to others.
What changes
  • Set LITELLM_BUDGET_MODE=enforce
  • Push max_budget: cents/100 AND soft_budget: 80% both
  • One-shot backfill re-run to apply to existing keys
  • Graceful error translation on remix / chat paths (never raw LiteLLM errors)
  • "Contact owner" UX when limit hit
NET EFFECT
Proxy blocks over-budget requests with a clean 4xx. UX shows "you've reached $X, ask team owner". Revert via env var if anything surprises us.
