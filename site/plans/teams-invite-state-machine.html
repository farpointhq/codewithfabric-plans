Invitation Acceptance — 5-State Migration (already built)
Source: src/lib/team-invite.ts::executeTeamMigration. Runs in a single prisma.$transaction. checkAntiHijacking prevents circular hierarchies.
NO_TEAM
Invitee has no owned team and no existing membership. Most common path for fresh signups.
Action TEAM_OWNER add TeamMember row with billingProvider: TEAM_OWNER.
OWNS_EMPTY_NO_SUB
Invitee owns a team but it has no members and no active subscription. (Will be the default state for every user once gap #3 ships.)
Action DELETE their empty team, TEAM_OWNER add as member of new team.
OWNS_EMPTY_WITH_SUB
Invitee owns an empty team that has its own active billing (they paid for themselves).
Action Keep their team, SELF add to new team with billingProvider: SELF.
OWNS_TEAM_WITH_MEMBERS
Invitee owns a real team with members — this is the "downline" case Ryan described.
Action HIERARCHY set their team's parentTeamId to the new team, TEAM_OWNER add as member.
MEMBER_OF_OTHER
Invitee is already a member of some other team (not owner).
Action LEAVE old TeamMember row, TEAM_OWNER add as member of new team.
SELFmember pays own billing
TEAM_OWNERteam owner pays for this member
HIERARCHYinvitee's team becomes a sub-team (downline)
DELETE / LEAVEprior team state is removed
All 5 states are already handled on main with anti-hijacking protection and a hard depth limit (MAX_HIERARCHY_DEPTH). No schema or migration-logic changes are needed from Ryan's vision — only a guard at the invite-creation site (api/team/invitations/route.ts) to enforce the "inviter must have Unlimited" rule, which is Gap #4.
